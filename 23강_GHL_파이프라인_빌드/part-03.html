예(A 푸르트): "3~7세 자녀를 둔 엄마에게 정기 구독 고객이 되도록, 신규 DM → 상담 → 결제 → 수령 → 재구매 리마인드 흐름으로 움직인다."

실무 표준 5단계에서 이름만 팀 클라이언트에 맞게 바꿔 주시면 됩니다.

단계 수는 3~5단계 권장. 너무 많으면 카드 이동이 번거로워집니다[^G2].

단계 이름은 동사 중심으로 쓰시면 실무에서 기준이 뚜렷해져요. 예: "상담" → "첫 상담 완료".

네 가지 질문이면 · 충분하겠어요?

  • 01 📉 설계 · 복잡할수록 늘어지기 →
    질문이 많아질수록 팀 토론이 늘어집니다. 결정 점이 많으면 합의가 어려워지는 흔한 현상이에요.
  • 02 🎯 네 질문 · 꼭 필요한 것만 →
    아키텍처 문서에 꼭 필요한 네 질문만 남기면, 팀이 답을 채우는 과정 자체가 설계가 됩니다.
  • 03 📝 질문의 답 · 그대로 문서 →
    오늘은 "설계하기"가 아니라 "질문에 답하기". 네 답변을 이어 붙이면 아키텍처 문서 한 장이 나옵니다.

📝 강사 노트

오늘 팀이 2~3개 만 확정하시면 됩니다. 나머지는 24·25강에서 채워요.

오늘 설계할 Workflow는 한 개입니다. "폼 제출 → 태그 추가 → 24시간 뒤 SMS 발송" 정도의 단순 3노드면 충분해요.

  • 02 🗺️ Stages · 단계 3~5개 →
    표준 5단계(신규문의→상담→결제→납품→재구매)에서 이름만 팀 클라이언트에 맞춰 바꿉니다. 동사 중심이 기준이 뚜렷해요.
  • 03 🏷️ Tags · 세 덩어리 2~3개 →
    관심 분야·응대 단계·가치 구간 세 덩어리에서 오늘은 2~3개만 확정. 나머지는 24·25강에서 채워요.
  • 04 ⚡ Automation · Workflow 1개 →
    "폼 제출 → 태그 추가 → 24h 뒤 SMS" 같은 단순 3노드. 트리거 1개·액션 1~2개·대기 1개면 충분합니다.
