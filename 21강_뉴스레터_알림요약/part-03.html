📝 강사 노트
💬 그대로 읽으세요
뉴스레터(Newsletter)는 "정해진 주기로 구독자에게 발송되는 이메일 묶음"을 뜻합니다[^1]. GHL에서는 이걸 두 가지 경로로 만들 수 있어요. 헷갈리기 쉬운 지점이라 먼저 구분하겠습니다.

경로 A · 이메일 캠페인(Email Campaigns): Marketing → Email → Campaigns에서 만드는 1회성 또는 일정 예약 발송 단위예요. 세그먼트(Segment · 조건으로 잘라낸 수신자 그룹)를 수신자로 지정하고, 즉시 또는 지정 일시에 발송합니다. 월간 뉴스레터처럼 "매달 첫째 주 월요일" 같은 반복 일정은 이 경로를 씁니다[^2].

경로 B · 워크플로우(Workflow) 내 이메일 액션: Automation → Workflows에서 "조건 → 동작" 흐름 중 한 단계로 이메일을 보내는 방식이에요. "폼 제출 시 즉시 환영 메일" 같은 트리거 기반 1회 발송에 적합합니다.

오늘 우리가 만드는 건 경로 A입니다. 경로 B는 18강·19강에서 이미 다루었으니 오늘은 건너뛸게요.

뉴스레터를 전체 리드에게 뿌리는 것이 아니라, 지금 단계에 맞는 메시지가 필요한 리드에게만 보내는 것이 실무의 기본기예요. 예를 들어:

✅ 보낼 곳: `need-nurturing`(정보 수집 단계) + `high-intent`(높은 의향) 태그 보유자 → 관계 유지 메시지가 도움 됨

❌ 보내지 말 곳: `budget-low`에 이미 20강 Reactivation 캠페인 대상이 된 리드 → 중복 피로 방지

내부 알림은 고객에게 가는 메시지가 아니라 담당자(나 또는 팀)에게 가는 메시지입니다. 리드 한 명의 행동 중 우리가 놓치면 안 되는 순간만 골라 즉시 알려 주는 용도예요.

"웹훅" · 어려운 단어인가요?

  • 01 🌱 처음이어도 · 한 줄 개념 →
    "내 시스템에 일이 생기면 · 네가 정한 URL로 · 그 요약을 JSON으로 자동 전송." 이 한 줄이 웹훅(Webhook) 전체예요.
  • 02 📮 비유 · 자동 배달원 1명 →
    GHL에 "이 주소로 이 쪽지 전달"이라고 적힌 자동 배달원을 하나 둔 것과 같아요. 이벤트가 생기면 배달원이 즉시 출발합니다.
  • 03 🔑 핵심 · 두 URL 아닌 한 URL →
    Slack이 발급한 Incoming Webhook URL 한 개를 GHL Outbound Webhook 목적지로 등록합니다. "양 끝"이 같은 URL이에요.

Newsletter · 두 경로 분기

  • 01 📬 Newsletter · 정기 이메일 묶음 →
    뉴스레터(Newsletter · 정해진 주기로 구독자에게 발송되는 이메일 묶음). 오늘 만들 것은 "월 1회 반복 발송"입니다.
  • 02 🅰️ 경로 A · Email Campaigns →
    Marketing → Email → Campaigns. 세그먼트(Segment · 조건으로 잘라 낸 수신자 그룹) 대상 1회성 또는 예약 발송 단위. 오늘 경로가 이쪽입니다.
  • 03 🅱️ 경로 B · Workflow 이메일 액션 →
    Automation → Workflows. "폼 제출 시 즉시 환영 메일" 같은 트리거 기반 1회 발송. 18~19강에서 다뤘으니 오늘은 건너뜁니다.
  • 04 🎯 세그먼트 · 지금 단계에 맞게 →
    전체 발송 대신 `need-nurturing`+`high-intent` 대상만. 20강 Reactivation 대상인 `budget-low`는 중복 피로 방지로 제외합니다.
CONCEPT · 01
📝 강사 노트
💬 그대로 읽으세요
이메일: 담당자 이메일로 직접 발송 (가장 단순)

SMS: 담당자 휴대폰으로 문자 발송 (국내 발송은 통신사 승인 필요 · 오늘은 개념만)

Slack: 팀 메시지 채널에 자동 게시 (오늘의 실습 대상)

웹훅(Webhook): 외부 시스템으로 데이터 전송 (Slack을 연결하는 기술적 다리)

공공 강의에서 "알림 채널" 하면 가장 현실적인 조합은 Slack + 웹훅입니다. 외부 시스템 승인 없이 수업 안에서 작동 확인까지 가능하거든요.

Internal Notification · 4채널

  • 01 📧 이메일 · 담당자 직메 가장 단순 →
    담당자 이메일로 직접 발송. 가장 단순하지만 실시간성이 낮아 "지금 당장" 반응용으로는 부족합니다.
  • 02 📱 SMS · 국내 통신사 승인 →
    담당자 휴대폰 문자. 국내 발송은 통신사 사전 승인이 필요해 오늘은 개념만 짚고 실제 구성에선 제외합니다.
  • 03 💬 Slack · 팀 채널 자동 게시 →
    팀 Slack 채널에 자동 메시지. 즉시성·가독성·검색성까지 실무 요건을 가장 균형 있게 만족시켜 오늘의 선택이 됩니다.
  • 04 🪝 웹훅 · Slack 연결용 기술 다리 →
    페이로드(Payload · 웹훅이 실어 보내는 데이터 묶음)를 외부 시스템으로 전달. 오늘은 Slack을 붙이는 "기술 다리"로 씁니다.

[Slack 서버] └─ 위 URL 수신 → 지정 채널에 메시지 자동 게시

05 / 07

Outbound ↔ Incoming · 같은 URL

  • 01 📤 Outbound · GHL 나가는 쪽 →
    Outbound Webhook(나가는 웹훅 · GHL 워크플로우에서 외부로 데이터를 보내는 동작). Action 메뉴에서 목적지 URL 지정.
  • 02 📥 Incoming · Slack 받는 쪽 →
    Incoming Webhook(들어오는 웹훅 · Slack 앱 설정에서 발급하는 공식 수신 URL). 채널 하나에 연결되어 메시지를 자동 게시합니다.
  • 03 🔗 한 URL · 두 각도 →
    두 URL이 다른 게 아니에요. Slack이 발급한 Incoming URL을 GHL Outbound 목적지로 등록 — 같은 URL을 양 끝에서 부르는 구조입니다.
  • 04 🗺️ 관계도 · 두 장치 + 태그 풀 →
    폼 → 태깅 엔진(17강) → 위쪽 가지 뉴스레터(경로 A) · 아래쪽 가지 Slack 알림(Workflow→Outbound→Slack URL). 한 지도에 담습니다.
CONCEPT · 03
📝 강사 노트
💬 그대로 읽으세요
이 흐름에서 두 구간을 구별하는 것이 중요해요.

Outbound Webhook (나가는 웹훅): GHL이 보내는 쪽. 워크플로우 동작(Action)으로 설정[^4].

Incoming Webhook (들어오는 웹훅): Slack이 받는 쪽. Slack 앱 설정에서 공식 제공하는 공개 URL[^5].

두 URL이 같은 것을 서로 다른 각도에서 부르는 것이 아니라, Slack이 발급해 준 Incoming Webhook URL을 GHL의 Outbound Webhook 목적지로 등록하는 구조입니다.

한 그림으로 정리하면 이렇게 됩니다.

Part 03 · 네 줄 정리

  • 01 📬 뉴스레터 = Email Campaigns →
    오늘 만들 뉴스레터는 Marketing → Email → Campaigns 경로. 세그먼트에 예약 발송하는 단위입니다.
  • 02 🔔 내부 알림 = 담당자용 신호 →
    고객이 아닌 담당자에게 가는 메시지. 오늘은 Slack 채널 수신을 선택합니다.
  • 03 🪝 웹훅 = 이벤트→URL 자동 전송 →
    이벤트 발생 시 외부 URL로 JSON 페이로드 전송. 오늘은 Slack에 붙이는 기술 다리 용도예요.
  • 04 🔗 양 끝 · 같은 URL 한 개 →
    Outbound(GHL)와 Incoming(Slack)은 "URL 한 개"의 양 끝. Slack 발급 URL을 GHL 목적지로 등록하는 구조입니다.
