공식 사전 실행 · 3단계

  • 01 📍 위치 · 빌더 우상단 →
    GHL 워크플로우 빌더를 열면 우상단에 `Test Workflow`(테스트 워크플로우 · 사전 실행 전용) 버튼이 있습니다.
  • 02 👤 Step 1 · TEST 컨택트 선택 →
    테스트 컨택트 선택 창에서 기존 TEST 컨택트 하나를 고릅니다. 분기마다 별도 TEST 컨택트 권장(공식 가이드).
  • 03 ▶️ Step 2 · Run Test 클릭 →
    Run Test(테스트 실행) 클릭 → 각 노드의 통과·실패가 시각적으로 표시. 실 컨택트 리드에는 메시지가 가지 않습니다.
  • 04 ⚠️ 주의 · 반복 시 정확도 저하 →
    같은 컨택트로 여러 번 반복 테스트하면 정확도가 떨어진다고 공식 문서가 명시. 분기별로 다른 TEST 컨택트 준비가 권고안.

📝 강사 노트
💬 그대로 읽으세요
1. 테스트 컨택트 선택: 기존 컨택트 목록에서 TEST 컨택트를 하나 고릅니다. 2. `Run Test`(테스트 실행) 클릭: 선택한 컨택트 정보로 워크플로우가 그 자리에서 실행됩니다. 3. 실행 결과 확인: 각 노드의 통과·실패 여부가 시각적으로 표시됩니다.

여기서 가장 흔히 놓치는 함정 하나. `Test Workflow` 버튼으로 실행한 테스트는 wait timer(대기 타이머 · 워크플로우 중간에 "N분 뒤"·"N일 뒤" 지연하는 설정)를 스킵하거나 초 단위로 압축합니다[^T2-2].

04 / 07

테스트 vs · 실제 트리거

  • 01 ⏱️ wait timer · 스킵 vs 실제 →
    Test Workflow는 wait timer(대기 타이머 · "N분·N일 뒤" 지연 설정)를 스킵 또는 초 단위 압축. 실제 트리거는 설정대로 대기.
  • 02 👤 컨택트 · 수동 vs 자동 진입 →
    사전 실행은 컨택트를 수동 선택. 라이브는 실제 이벤트(폼 제출·태그 부착)로 컨택트가 자동 진입.
  • 03 🔍 발견 가능 버그 · 범위 차이 →
    사전: 논리·분기·메시지 내용. 실제: 시간 지연 관련 전체 포함. 둘을 순서대로 쓰라는 게 공식 권고.
  • 04 🕰️ 오늘 수업 · 사전 중심 →
    수업 시간상 `Test Workflow` 중심. 시간 지연 버그 확인은 "오늘 실행 + 내일 결과 확인"으로 이관합니다.

Part 02 · 세 줄 정리

  • 01 🔇 조용한 실패 · 쌓인다 →
    자동화 실패는 티를 내지 않고 쌓입니다. 중복·엉뚱한 컨택트·트리거 미발동·분기 오류 — 라이브 전 테스트가 필수인 이유.
  • 02 🎛️ Test Workflow · 공식 버튼 →
    GHL 빌더 우상단 `Test Workflow` = 공식 사전 실행. Run Test로 노드별 통과·실패 시각 표시.
  • 03 ⏱️ wait timer · 스킵 주의 →
    사전 실행은 wait timer 스킵 → 시간 지연 버그는 실제 트리거 테스트에서만 잡힘. 두 모드를 순서대로 사용.
