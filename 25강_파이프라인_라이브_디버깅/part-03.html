에러를 만났습니다 · 어디부터 · 손대시나요?

  • 01 🤷 흔한 장면 · 이것저것 수정 →
    테스트에서 실패 발견 → 곧바로 빌더 열고 이것저것 고쳐 보는 그림, 낯익지 않으신가요? 결국 "왜 고쳐졌는지 모르는" 상태가 됩니다.
  • 02 🎲 우연히 · 고쳐지는 위험 →
    한 번에 두 가지를 고치면 무엇이 진짜 원인이었는지 놓칩니다. 같은 오류가 다음 주에 또 터지면 그때도 우연에 기대게 됩니다.
  • 03 🧭 순서가 있다 · 4단계 틀 →
    재현 → 로그 → 가설 → 수정. 이 네 단계가 에이전시 실무의 공통 디버깅 사고법. 오늘 틀을 잡아 둡니다.

03 / 07

팀 공통 · 다섯 시나리오

  • 01 🏷️ 리드 제출 → 태깅 →
    신규 리드 제출 시 VIP·관심·잠재 중 올바른 태그가 자동 부착되는가. 자주 깨지는 지점 — 조건식 AND/OR 혼동, 태그명 대소문자·공백.
  • 02 📅 예약 발행 스케줄 →
    24강에서 병합한 예약 콘텐츠가 지정 요일·시간대에 발송 큐에 걸리는가. 자주 깨지는 지점 — 타임존 누락, 빈 슬롯, 채널 연동 만료.
  • 03 💬 자동 DM · 분기 →
    예약·환불·일반 유형에 따라 DM 경로가 다르게 분기되는가. 문의 유형 필드를 세 번 바꿔 실행. 자주 깨지는 지점 — else 경로 누락.
  • 04 🔔 재활성화 · Slack 알림 →
    "마지막 활동" 91일 전 리드가 재활성화 세그먼트 자동 편입되는가 + VIP 부착 시 Slack `#알림` 채널로 요약이 도착하는가.

04 / 07

재현 → 로그 → 가설 → 수정

  • 01 🔁 1단계 · 재현 · 같은 조건 →
    같은 조건으로 한 번 더 실행. Claude Code는 `/resume`(슬래시 리줌 · 과거 세션 재개)으로 직전 세션 복원 · GHL은 Test Workflow 재실행.
  • 02 🔍 2단계 · 로그 · 실패 노드 →
    GHL은 Automation → Executions(실행 내역) 창에서 실패 노드 확인. Claude Code는 `--debug`(상세 로그 모드) 플래그로 API·도구·타이밍 상세 로그.
  • 03 💡 3단계 · 가설 · 플랜 모드 →
    Plan Mode 진입(`/plan` 또는 Shift+Tab 두 번) → Claude가 수정 계획만 읽기 전용으로 제시. "원인 후보 1·2·3" 한 문장씩 적기.
  • 04 🔧 4단계 · 수정 · 한 가지만 →
    가장 검증 쉬운 가설부터 한 번에 한 가지만 고치고 1단계 재현으로 돌아갑니다. 한 번에 두 가지 고치면 원인을 영원히 놓칩니다.
