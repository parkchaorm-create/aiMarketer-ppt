💬 그대로 읽으세요
실습 A에서는 각 팀이 선택한 가상 클라이언트 기준으로 5개 표준 테스트 케이스 중 최소 3개를 실행하고, 아래 양식의 결과 기록표 1장을 완성합니다. 이 표가 오늘의 핵심 산출물 중 하나입니다.

1. TEST 컨택트 준비: GHL → Contacts → Add Contact → 이름 `TEST_{본인이름}` · 본인 번호·이메일. 2. 첫 번째 케이스 선택: Part 03의 5개 중 우선순위 높은 케이스 하나. 3. 컨택트 속성 조정: 해당 케이스의 조건에 맞게 커스텀 필드 설정 (예: 주문 이력·문의 유형). 4. `Test Workflow` 실행: 대상 워크플로우 열고 우상단 버튼 → `Run Test` → TEST 컨택트 선택. 5. 결과 기록: 아래 양식에 1줄 작성. 6. 다음 케이스 반복 (최소 3개). 7. `/memory` 캡처: Claude Code 터미널에서 `/memory` 실행 → 22강에서 만든 `ghl-deploy.md` 가 로드되는지 확인.

  • 01 🏃 조급함 · 실 발송 사고 →
    세팅 없이 바로 Test Workflow를 누르면, 일부 상황에서 실 세그먼트까지 메시지가 흐를 수 있어요. 5분 세팅이 사고 하나를 막습니다.

    테스트만 돌리고 기록을 안 남기면 "돌렸는데 뭐가 됐더라"로 끝납니다. 오늘의 산출물은 "결과 기록표 1장".

    TEST 컨택트 · 기록표 · `/memory` 확인 · 업로드 포맷. 네 가지만 먼저 챙기면 실행이 매우 깔끔해집니다.

  • 01 ✅ 팀 단위 · 기록표 1장 완성 →
    팀 단위로 결과 기록표 1장이 완성되어 있는가. 개인별이 아니라 팀 단위 단일 문서여야 인수인계가 쉽습니다.
  • 02 ✅ 최소 3 케이스 · ✅/❌ 명확 →
    5개 중 최소 3개가 실행되었고 성공·실패 마크가 명확한가. 완성도보다 "일부러 돌려 본 경험"이 합격선.
  • 03 ✅ 실패 1개 · 원인 한 줄 →
    실패 케이스 1개 이상에 "의심되는 원인 한 줄"이 적혀 있는가. 이게 Part 06 라이브 디버깅의 출발점입니다.
