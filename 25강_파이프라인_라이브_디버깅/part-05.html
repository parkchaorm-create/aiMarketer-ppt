실습 B는 만드는 실습이 아니라 관전·기록 실습입니다. 강사가 대표 팀 중 한 팀의 실패 케이스 하나를 골라 4단계 사고법을 소리 내어 시연합니다. 수강생은 각 단계마다 내 파이프라인에 적용할 한 줄을 기록합니다. 이 기록이 두 번째 산출물입니다.

1. 대표 팀 선정: 강사가 Part 04 결과 중 재현 가능한 실패 사례를 하나 지목. 해당 팀의 화면을 대형 화면에 공유. 2. 1단계 · 재현: 강사가 `Test Workflow` 재실행하며 "지금 1단계입니다" 라벨 멘션. 수강생은 메모 1행 작성. 3. 2단계 · 로그 확인: GHL Executions 창에서 실패 노드 확인. Claude Code `--debug` 로그도 병행. 멘션 후 메모 2행. 4. 3단계 · 가설 수립: Claude Code에서 `/plan` 진입(또는 Shift+Tab 2회). Claude가 수정 계획을 읽기 전용으로 제시. 멘션 후 메모 3행. 5. 4단계 · 수정·재실행: 일반 모드 복귀 → 가설 1개만 수정 → 1단계 재현. 멘션 후 메모 4행. 6. 결과 확인 · 수강생 질문 1~2개.

세 가지 · 막힘 · 해결

  • 01 👻 Test Workflow · 버튼이 안 보임 →
    워크플로우가 Draft(드래프트 · 초안) 상태인지 확인. Published(게시됨) 상태여야 `Test Workflow` 버튼이 우상단에 표시됩니다.
  • 02 🔁 반복 실행 · 결과가 이상함 →
    같은 TEST 컨택트로 반복하면 정확도 저하(공식 권고). 분기별로 다른 TEST 컨택트(TEST_예약·TEST_환불 등)를 준비.
  • 03 ⏰ 예약 시간 · 안 잡힘 →
    타임존 설정이 계정·워크플로우 두 곳에 모두 정확한지 확인. 한 곳만 맞춰 놓으면 큐 등록이 누락됩니다.
  • 04 🆘 그 외 · 슬랙 강사-QnA →
    위 세 가지에 해당하지 않는 막힘은 슬랙 `#강사-QnA` 캡처와 함께 질문. 강사가 1차 확인 후 답변 드립니다.
